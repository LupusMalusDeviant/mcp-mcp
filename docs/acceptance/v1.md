> [!WARNING]
> **Veraltet — Zeitkapsel.** Diese Abnahme hakte alle Kriterien ab, bevor ein unabhängiger
> Abgleich Anforderung↔Code lief; der fand später 23 Lücken, darunter einen Redaction-Defekt
> (Zugangsdaten im Klartext im Audit-Log) und einen Lasttest, der in Wahrheit nie ausgeführt wurde.

## Keyfeatures (ursprüngliches Briefing)

| Keyfeature | Umsetzung | WP |
|---|---|---|
| 1. Hot-swappable MCP-Server | Supervisor mit Crash-Restart, `list_changed` | WP1/WP4 |
| 2. API ↔ MCP (beide Richtungen) | REST-Fassade + OpenAPI-Import als virtueller Upstream | WP5 |
| 3. Token sparen | Profile (Pinned + Lazy-Meta-Tools), ≥ 80 % belegt | WP2 |
| 4. Logging (Wer/Was/Wie) | Audit-Pipeline mit Redaction, Filter, Export | WP3 |
| 5. RBAC | Default-Deny, Server→Tool-Vererbung, Rate-Limits | WP2 |
| 6. Web-UI | Blazor: Dashboard, Server, Tools, RBAC, Logs, Token-Cockpit | WP6 |
| 7. Multi-Agent + zentrale Steuerung | Sessions pro Identität, Federation, Asset-/Skill-Verwaltung | WP4/WP6/WP7 ⚠️ |

> ⚠️ **Korrektur (2026-07-18):** Keyfeature 7 war zum Zeitpunkt dieser Abnahme **unvollständig** — Assets ließen sich zwar verwalten, wurden den Agenten aber nicht ausgeliefert (FR-40). Ebenso fehlten die Rollback-UI (WP6.2), der Description-Override (FR-14) und der Metriken-Export (FR-26), obwohl die Pakete als erledigt markiert waren. Alle vier Punkte sind inzwischen umgesetzt und getestet; Details im [Plan-Änderungslog](../plans/0001-mcp-mcp-meta-gateway.md). Dieses Dokument bleibt als Zeitkapsel der damaligen (teils zu optimistischen) Abnahme stehen.

## Weitere DoD-Punkte (WP7)

- **Docker-Image < 300 MB:** chiseled-Runtime; in CI (`docker`-Job) gebaut, Größe geprüft, Container-Smoke gegen `/healthz`.
- **Security-Audit ohne offene High/Critical:** WP7.2-Audit — kein Critical, kein echtes High; alle umsetzbaren Medium/Low-Findings behoben ([Threat-Model](../security/threat-model.md)).
- **Federation (FR-05):** `FederationTests` — zweiter Gateway als Upstream + Self-Loop-Erkennung (508).

## Bewusste Abweichungen / offene Punkte für v1.1

- **(*) NFR-01-Referenzmessung:** Die harte 50-ms-p95-Schranke gilt für Referenz-Hardware; auf den (teils 2-Kern-)CI-Runnern läuft der Test als Smoke mit relaxter Schranke. ✅ **In v1.1 formal nachgeholt:** 20 Sessions / 100 In-Flight-Calls / 100-Tool-Katalog ergaben **p95 = 7,3 ms** (Calls) und **59,1 ms** (`tools/list`) bei 0 Fehlern — siehe [performance.md](performance.md).
- **EF-Migrations-Baseline:** v1.0 nutzt `EnsureCreated` (Schema ist frisch, funktional korrekt auf beiden Providern). ✅ **In v1.1 nachgezogen:** zwei provider-spezifische Migrations-Assemblies + `DatabaseInitializer` mit Baseline-Stamping; das Upgrade bestehender v1.0-Datenbanken läuft ohne manuellen Schritt und ohne Datenverlust (Tests gegen SQLite und PostgreSQL, siehe [operations.md](../operations.md#schema--upgrades)).
- **DataProtection-Key-at-rest-Schutz (KMS/X509)** und **PBKDF2-Iterationen auf 600k**: ✅ **beide in v1.1 umgesetzt** (X509-Key-Ring-Schutz optional per Env, PBKDF2 auf 600k mit abwärtskompatibler Verifikation) — siehe [Threat-Model](../security/threat-model.md).
