# Abnahme (unveröffentlicht, Stand `main`) — MCP-MCP Gateway

Nachweis der sieben Abnahmekriterien aus [PRD 0001, Abschnitt 8](../prd/0001-mcp-mcp-meta-gateway.md)
gegen den **tatsächlichen Code-Stand**, nicht gegen Plan-Häkchen. Jedes Kriterium ist an einen
konkreten, benannten Test gebunden.

**Stand: 2026-07-20. Testbestand: 289 Testfälle** (216 Testmethoden; `[Theory]`-Fälle expandiert),
grün auf Windows **und** Linux. Persistenz läuft zusätzlich gegen echtes PostgreSQL — in CI auf
Linux **erzwungen** (`MCPMCP_REQUIRE_POSTGRES=1`), damit ein stiller Skip nicht als grün durchgeht.
Zeilenabdeckung `McpMcp.Core` ≥ 80 % ist als CI-Gate verankert (zuletzt gemessen 86,75 %).

> **Warum dieses Dokument die frühere [v1.md](v1.md) ersetzt.** Die v1.0-Abnahme hakte alle
> Kriterien ab, bevor ein unabhängiger Abgleich Anforderung↔Code lief. Der fand **23 Lücken** —
> darunter ein Redaction-Defekt, über den Zugangsdaten im Klartext ins Audit-Log gelangten, und
> einen Lasttest, der über eine nie gesetzte Umgebungsvariable **permanent übersprungen** wurde,
> während die Suite grün meldete. Wegen des Redaction-Defekts sind die Releases **v1.0.0 und
> v1.1.0 zurückgezogen**. v1.md bleibt als Zeitkapsel der damals zu optimistischen Abnahme stehen;
> maßgeblich ist dieses Dokument.

**nirgends**, weil `MCPMCP_RUN_BENCHMARK` in keiner CI-Ausführung gesetzt war. Er läuft jetzt im
Job `benchmark` und ist nachweislich ausgeführt worden:

| `tools/list` p95 (100 Tools) | 59,1 ms | 118,0 ms | ≤ 200 ms ✅ |
| Fehler unter Last | 0 | 0 | 0 ✅ |

## Guardrail — Secret-Erkennung (über das Briefing hinaus)

Nach den sieben Kriterien hinzugekommen ([ADR-0011](../adr/0011-secret-erkennung-als-guardrail.md)):
Der Gateway prüft Tool-Argumente und -Ergebnisse auf Zugangsdaten und blockiert Treffer.

- **Kern:** `SecretGuardTests` (23 Fälle) — bekannte Muster erkannt; Git-SHAs, UUIDs, MD5-Hashes,
  Stripe-*Public*-Keys bewusst **nicht** getroffen; Befund enthält nie den Klartext; ReDoS-Muster
  hängt nicht; nicht unterstützte Regex-Konstrukte werden beim Speichern abgelehnt; Latenz < 1 ms.
- **Pipeline:** `GuardedInvocationTests` — ausgehend blockiert vor dem Upstream (`Denied`),
  eingehend nach dem bereits ausgeführten Call (`GuardBlocked`, mit sprechender „nicht
  wiederholen"-Meldung).
- **Live verifiziert:** eine echte `.env` mit Fake-GitHub-Token durch den Filesystem-Server gelesen
  → `GuardBlocked`; der Token erschien **nicht** in der API-Antwort, **nicht** im Audit-Log,
  **nicht** im Server-Log.

## Weitere DoD-Punkte

- **Docker < 300 MB, non-root, arm64:** im `docker`-CI-Job gebaut, Größe geprüft, Container-Smoke
  gegen `/healthz`; **linux/arm64** wird per `buildx` tatsächlich gebaut (NFR-05); beide
  Compose-Beispiele werden per `docker compose config` validiert.
- **Security-Audit:** kein Critical/High ([Threat-Model](../security/threat-model.md)). Der später
  gefundene Redaction-Defekt und die begrenzte Loop-Erkennung sind dort ergänzt.
- **MCP-Protokollversion** (NFR-09): per Handshake-Test auf „2025-06-18 oder neuer" festgenagelt,
  hängt nicht mehr still am SDK.

## Bewusst offen

- **FR-04 (Sampling/Elicitation durchreichen), Kann-Teil:** nicht umgesetzt, als Abweichung
  dokumentiert ([ADR-0010](../adr/0010-sampling-elicitation-nicht-durchreichen.md)) — eine
  server-initiierte Anfrage lässt sich keinem konkreten Downstream-Agenten zuordnen.
- **FR-02 (SSE-Legacy-Transport):** upstream über den SDK-AutoDetect-Fallback unterstützt,
  downstream bewusst nur Streamable HTTP ([ADR-0009](../adr/0009-sse-legacy-transport.md)).
- **FR-20 (Webhooks), FR-32 (Approval-Flows):** Kann-Anforderungen, begründet auf v2 vertagt.
- **FR-21:** geloggt wird die *Rolle* des Aufrufers; das zugeordnete *Profil* ist nicht Teil des
  Audit-Datensatzes. Der PRD-Wortlaut „Profil/Rolle" lässt beides zu; die Wahl ist bewusst.

## Noch keine Betriebshärte

Der Gateway ist gründlich getestet, aber **nie produktiv betrieben**: keine Uptime über Tage, keine
Datenbank jenseits von Testgrößen. Der Live-Test gegen einen echten MCP-Server (oben) ist der erste
Kontakt mit fremdem Code und verlief sauber — er ersetzt keinen Dauerbetrieb. Vor einem Release ist
das die einzige verbleibende Unbekannte, und sie liegt außerhalb dessen, was Tests zeigen können.
